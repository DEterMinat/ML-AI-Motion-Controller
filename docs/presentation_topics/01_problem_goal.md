# 01) Problem & Goal

## ภาพรวมปัญหาที่โปรเจกต์แก้
โปรเจกต์นี้แก้ปัญหา **การควบคุมเกมด้วยคีย์บอร์ด/เมาส์ที่ไม่เป็นธรรมชาติ** สำหรับเกมแนวต่อสู้ โดยเฉพาะใน Roblox ที่ผู้เล่นต้องกดคอมโบหลายปุ่มเร็วมาก และผู้เล่นใหม่มักติดเรื่อง timing, key mapping และ cognitive load

ปัญหาเชิงผู้ใช้ที่พบจริง:
1. ต้องจำคีย์จำนวนมาก (โจมตี/หลบ/ป้องกัน/คอมโบ)
2. การกดคอมโบพร้อมกันเสี่ยงพลาดสูง
3. ผู้เล่นใหม่เข้าถึงยาก และเหนื่อยจากการควบคุมมากกว่ากลยุทธ์เกม
4. ประสบการณ์การเล่นไม่ immersive เพราะร่างกายไม่ได้สัมพันธ์กับ action ในเกม

โปรเจกต์จึงสร้าง **Motion Controller** ที่แปลงท่าร่างกายเป็น action ในเกมโดยตรง:
- ผู้เล่นชกจริง → ตัวละครชก
- ผู้เล่นเอียงหลบจริง → ตัวละครหลบ
- ผู้เล่นยกการ์ดจริง → ตัวละครบล็อก

---

## ทำไมต้องใช้ Motion Controller กับ Roblox

### เหตุผลเชิงผลิตภัณฑ์
- เพิ่มความ immersive และ engagement ของผู้เล่น
- ลด learning curve สำหรับผู้เล่นใหม่ (ท่าทางเข้าใจง่ายกว่าจำปุ่ม)
- แตกต่างจากระบบควบคุมทั่วไป (จุดขายเกม/ระบบ)

### เหตุผลเชิงเทคนิค
- Roblox รองรับการเชื่อมต่อข้อมูลภายนอกผ่าน middleware ได้
- ระบบนี้มีทั้งช่องทาง **Direct Input** (กดปุ่มแทนผู้เล่น) และ **WebSocket** (`ws://localhost:8765`) สำหรับเชื่อมกับ game client
- Pipeline ปัจจุบันทำงานแบบ real-time ได้บน CPU ทั่วไป

---

## Goal หลักของโปรเจกต์

### Goal ระดับระบบ
1. ตรวจจับ action จาก webcam แบบ real-time
2. แมป action ไปเป็นคำสั่งเกมที่ใช้งานได้จริง
3. ลด false trigger ระหว่างเล่นจริง
4. ทำให้ระบบพร้อม deploy ในสภาพแวดล้อมผู้ใช้จริง

### Goal ระดับโมเดล
1. Accuracy สูงพอสำหรับ gameplay (>=95%)
2. แยก 9 action classes ได้ครบถ้วน
3. ไม่มี data leakage ในขั้นประเมินผล
4. รองรับความผันผวนของ pose ด้วย temporal smoothing + motion analyzer

## KPI หลัก (แนะนำใช้ในสไลด์)

| KPI | Target | Current Status (จากผลล่าสุด) | หมายเหตุ |
|---|---:|---:|---|
| Offline Test Accuracy (MLP) | >= 95% | **99.54%** | จาก `training/executed/executed_train.ipynb` |
| ครบจำนวนคลาสใน Test Set | 9/9 | **9/9** | หลังแก้ stratified video-based split |
| Robustness ต่อ noise/variation | สูงขึ้นชัดเจน | ผ่าน | ใช้ augmentation + motion analyzer |
| Runtime Response | ลื่นสำหรับเกม | ผ่านเชิงสถาปัตยกรรม | GUI update loop + target FPS 30 |
| Usability | ตั้งค่าได้ใน UI เดียว | ผ่าน | มี confidence slider, cooldown slider, profiles |

> หมายเหตุ: KPI latency ควรวัดจริงหน้างาน (เครื่อง target + Roblox scene จริง) เพิ่มอีก 1 รอบก่อนใช้งาน production เต็มรูปแบบ

"โปรเจกต์นี้แก้ pain point เรื่องการคุมเกมที่ต้องกดปุ่มซับซ้อน โดยเปลี่ยนเป็นควบคุมด้วยท่าทางจริงผ่าน webcam สำหรับ Roblox Boxing โดยเราตั้ง KPI ว่าต้องแม่น, ลื่น, และใช้งานง่าย ผลล่าสุด MLP ได้ 99.54% บนการประเมินแบบไม่มี leakage และครอบคลุมครบ 9 คลาส ทำให้พร้อมต่อยอดสู่ deployment จริง"
